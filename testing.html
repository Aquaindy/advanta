What requires extra environment variables

Several features are wired to refuse cleanly (503 service_unavailable) when their secrets aren't set, instead of silently faking results. If you hit a 503 on one of these, that's expected — log it as Skipped + note "needs env config".

  • Google OAuth login — needs GOOGLE_LOGIN_CLIENT_ID, GOOGLE_LOGIN_CLIENT_SECRET, GOOGLE_LOGIN_REDIRECT_URI. Without these, /auth/google/start returns 503 and the "Continue with Google" button leads to an error redirect.
  • OpenAI / LLM features (content generation, AI Assistant, market intel, ICP/persona, refresh, ad creatives) — without OPENAI_API_KEY they return a clearly-labelled deterministic stub instead of an LLM result. Functionality still works for testing the flows.
  • Stripe billing — needs STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, STRIPE_PRICE_ID_*. Without these, /billing/checkout-session returns 503.
  • Connected ad / analytics platforms (Google Ads, Meta, LinkedIn, GA4, Search Console) — each requires its own client ID + secret. Without them, the "Connect" button returns 503 and integrations stay in disconnected state. UI flows up to that point still work.
  • Provider webhooks — need GOOGLE_ADS_WEBHOOK_SECRET / META_ADS_WEBHOOK_SECRET / LINKEDIN_ADS_WEBHOOK_SECRET. Tested via curl, not the UI.
  • Inbound email parsing — needs INBOUND_EMAIL_SECRET.
  • Public blog archive — needs MARKETING_WORKSPACE_SLUG set to one of your created workspaces' slugs to flip /blog from "coming soon" to a real archive.
  • Background workers — Celery is off by default in dev (WORKERS_ENABLED=false); tasks dispatch synchronously. Don't worry about Redis / worker processes for this pass.

3 · Two-factor authentication (TOTP)

Install a TOTP authenticator (Google Authenticator, 1Password, Authy) on your phone before starting.

TC-2FA-001 Enable 2FA — full enrollment lifecycle Functional

Steps

  1. Sign in as the Owner. Open /settings/profile.
  2. In the "Two-factor authentication" card, click Enable 2FA.
  3. The card now shows a base32 secret + an otpauth:// provisioning URI. In your authenticator app, scan or enter the secret.
  4. Type the 6-digit code from the app into the form and click Confirm and enable.

Expected result

  • The card flips to "TOTP-2FA is on. You'll be asked for a 6-digit code each sign-in."
  • An amber "Recovery codes" panel appears with exactly 8 codes. Copy these. They will not be shown again — only their SHA-256 hashes are stored.
TC-2FA-002 Login now requires 2FA code Functional

Steps

  1. Sign out.
  2. Sign in with email + password only (no OTP).
  3. Then sign in with email + password + the current 6-digit OTP from your app.

Expected result

The first attempt returns 401 with code two_factor_required — the form silently reveals an additional "2FA code" field. The second attempt succeeds.

TC-2FA-003 Wrong OTP rejected with friendly error Negative

  2. Try to sign in with email + password + a deliberately wrong code 000000.

Expected result

401 with code two_factor_invalid_code. Form shows "That 2FA code didn't work — try the next one." OTP field stays visible.

TC-2FA-004 Recovery code works exactly once Functional

Steps

  1. Sign out.
  2. Sign in using one of the recovery codes from TC-2FA-001 in place of the OTP.
  3. Sign out again.
  4. Try to sign in using the same recovery code a second time.

Expected result

First attempt: success. Second attempt: 401 two_factor_invalid_code — codes are single-use. You should still have 7 remaining recovery codes.

22 · Provider webhooks

TC-WHK-001 HMAC signature required Security

Pre-conditions

Set GOOGLE_ADS_WEBHOOK_SECRET=test-google-ads-secret in the API .env; restart.

Steps

  1. Run the curl below — this is a request without a signature.
curl -i -X POST http://localhost:8002/api/v1/provider-webhooks/google_ads \
  -H "Content-Type: application/json" \
  -d '{"event_type":"campaign.status_changed"}'

Expected result

HTTP 401 with detail "Missing webhook signature." A different response (200 or 200 ack) here would mean the auth gate is broken — flag it as a Fail.

TC-WHK-002 Valid HMAC accepted; unknown account 200-acked Functional

Steps

  1. Run the bash script below in a terminal.
SECRET="test-google-ads-secret"
BODY='{"event_type":"campaign.status_changed","account_id":"unknown-acct","campaign_id":"x","status":"paused"}'
SIG=$(echo -n "$BODY" | openssl dgst -sha256 -hmac "$SECRET" | sed 's/^.* //')
curl -i -X POST http://localhost:8002/api/v1/provider-webhooks/google_ads \
  -H "Content-Type: application/json" \
  -H "X-Provider-Signature: $SIG" \
  -d "$BODY"

Expected result

200 with body {"matched": false, "reason": "unrecognized_account"} — the webhook is authenticated but the account doesn't belong to any workspace, so the change is silently ignored (we don't want providers retrying forever).
