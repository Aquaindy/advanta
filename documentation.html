Documentation · v0.0.1

AdVanta — your growth command center.

AdVanta is an operating layer for marketing teams. Specialized AI agents read your real ad accounts, GA4, and website, surface wasted spend, draft creative, run A/B tests, and apply approved changes back to the platforms — with every action gated by a configurable approval flow and a permanent audit trail.

This is the user manual. If you're a developer integrating against the API, jump to API reference. If you've never seen the app, start with Quickstart — it gets you to your first agent run in under five minutes.

Approval-gated by default

No silent spend changes

Every recommendation is yours to approve or reject. When you flip on Autopilot, you set explicit guardrails — daily spend caps, % caps per change, conversion thresholds, and a stop-loss kill switch.

Real OAuth, real data

No demo content, ever

If a platform isn't connected, you see a Connect button — never a fake chart. Tokens are encrypted at rest with Fernet; provider mutations capture prior-state for one-click revert.

Twelve specialist agents

Plus a Master Orchestrator

Each agent owns one slice of growth (paid, SEO, conversion, content, tracking). The Master Orchestrator chains them by goal so a single click can audit the whole stack.

Production-grade plumbing

From day one

Workspace isolation, RBAC, 2FA, API keys, Stripe billing, plan-aware quotas, dollar-cost LLM tracking, audit log on every action — all in the base build, not a future tier.

Quickstart #

Five minutes from sign-up to your first AI-generated recommendation. Every step can be undone — feel free to mess around.

1 · Create an account

Open /register, enter your email and a password (≥ 8 characters). Or click Continue with Google if your admin has wired Google login.

2 · Create your workspace

You land on the workspace selector after registration. Click Create workspace — name it after your team or company. The first user becomes its Owner.

Why workspaces?
Every record in AdVanta — campaigns, drafts, recs, audit rows — belongs to one workspace. Members get role-gated access; data never leaks across workspaces. One person can belong to many workspaces with different roles in each.

3 · Complete onboarding

From the dashboard, open Onboarding. The wizard collects: business name, website, industry, target audience, offer, primary conversion goal, monthly ad budget, geographic targets, current ad platforms, brand voice, and 1–3 competitors. Every agent reads from this profile.

Click Generate Growth DNA Profile on the next step. You'll get scored funnel- and paid-ads-readiness, a 30-day plan, and a shortlist of campaigns to run first.

4 · Run your first agent

Open Agents in the sidebar. Click Run on Onboarding Insight — it returns instantly with gap-analysis recommendations. Or pick Master Orchestrator with the input payload {"goal": "research"} to chain Market Intelligence and ICP / Persona automatically.

5 · Review & approve

Open Recommendations. Each card shows a risk pill (low / medium / high), source agent, and Approve / Reject. Click into one, read the suggested action, then approve. Every action is captured in the audit log.

That's it.
You're ready to connect platforms (guide), configure Autopilot (guide), or write your first blog post (guide).

Workspaces & members #

A workspace is the top-level tenant — the unit of data isolation, billing, and permissions. Everything you create (campaigns, drafts, recommendations, A/B tests, audit rows, API keys, even uploaded images) belongs to exactly one workspace.

You can be a member of many workspaces with a different role in each. The workspace switcher at the top of the dashboard scopes every page to one workspace at a time.

Inviting members

Owners + Admins can invite via the OpenAPI route POST /workspaces/{id}/members/invite (a UI surface lives in workspace settings). The invitee receives a hashed single-use token in an email; once they accept, they appear in the members list with the role you assigned.

Invitations expire 7 days after they're issued. The token is SHA-256 hashed at rest; the plaintext exists only in the email.

The sole-Owner protection: you can't demote the last Owner of a workspace. Add a second Owner first, then transfer.

Five roles, ordered most → least privileged. Risk-level role gating is enforced on the recommendation approval path so high-impact actions can't slip through a junior account.

Agents #

An agent is a self-contained piece of growth expertise — paid ads, SEO, conversion, content, tracking — that reads your real workspace data, runs in a single transaction, and produces three kinds of output:

  • Tasks — every step the agent ran, with input + output payloads + timestamps. Visible on the run detail page.
  • Skill outputs — structured JSON the agent produced for downstream consumption (used by reports, the Orchestrator, and the dashboard).
  • Recommendations — human-facing items routed through the approval flow.

Every run is metered against your plan's AGENT_RUN quota and (when LLM calls happen) the LLM_CALL token meter. See plan limits.

See Agent reference for what each one reads, writes, and skips.

Recommendations & approvals #

Every visible suggestion in AdVanta is a recommendation with a clear lifecycle:

  • open — produced by an agent, awaiting human review.
  • approved — a member with the right role pressed Approve. If the rec carries an executable action (e.g. pause campaign, change budget), the change is applied to the connected provider as part of the same transaction.
  • rejected — explicitly dismissed; the reason is captured.
  • executed — provider-side change confirmed.
  • archived — soft-deleted; out of the active list.

Each approval / rejection creates an audit row tagged with the actor, IP, user-agent, and a JSON diff of what changed.

Editable fields

Admins can edit the user-facing copy (title, summary, expected_impact, suggested_action) on a recommendation before approving. Risk level + recommendation type are immutable — they're part of the agent's structured output.

Reverting an executed recommendation

Provider mutations capture the prior state on the way out. From the recommendation detail page, click Revert on the execution row — the previous state is replayed back to the platform. Audit-logged as a separate row with is_revert=true.

Three modes #

AdVanta runs in one of three modes per workspace. New workspaces default to Approval.

Advisor

Recommendations only

Agents analyze and propose. Nothing reaches a connected provider, ever. Use this when you want the insights without giving up control of any outbound writes.

Approval (default)

Recommendations + manual approve flow

Agents propose. A human approves. We write to the provider with prior-state captured. Every action is audit-logged.

Autopilot

With your guardrails

Auto-approves recommendations that pass every guardrail you've set (spend cap, % cap, conversion threshold, allowed action types, risk ceiling). Audit-logged as SYSTEM. One-click stop-loss flips the kill switch instantly.

Switching to Autopilot requires complete guardrails — the API refuses with a 422 if any of max_daily_spend_increase_cents, max_daily_spend_total_cents, max_pct_increase_per_change, or allowed_action_types is missing. See the configuration guide.

Audit log #

Every meaningful event in your workspace is captured. The audit log is append-only, owned by the workspace, and visible to Admins + Owners.

What gets logged

  • Approvals, rejections, edits
  • Provider executions + reverts (with prior-state diff)
  • Autopilot decisions (tagged SYSTEM)
  • Provider webhook events
  • API key mints + revokes
  • 2FA enable / confirm / disable
  • Member invitations + acceptances + role changes
  • Autopilot mode transitions
  • Content draft creates, edits, publishes

What's recorded per row

  • Actor type (USER / AGENT / SYSTEM) and id
  • Action (e.g. recommendation.approved, autopilot.approved, creative.updated)
  • Resource type + id
  • IP address + User-Agent
  • JSON metadata — typically the diff of what changed
  • Timestamp

Paid Ads #

Reads synced campaign data and surfaces actionable wasted-spend signals.

Reads

  • All campaigns in the workspace (any provider).
  • Campaign status, daily / lifetime budget, end date, provider.

Recommendation types it produces

  • paid_ads.no_campaigns (medium) — when no platforms are connected.
  • paid_ads.budget_unset (medium) — active campaigns with no budget set.
  • paid_ads.stale_active (high) — campaigns whose end_date has passed but status is still active.
  • paid_ads.single_platform (low) — concentration risk: 3+ active campaigns all on one platform.

Skips when

Never skips. With zero campaigns it surfaces a single "connect a platform" recommendation.

Content Writer #

Generates blog posts, landing-page copy, ad copy, meta descriptions, emails, or social posts. The output is saved as a ContentDraft row in draft status; nothing is auto-published.

Onboarding Insight #

The fastest agent. Reads your onboarding profile + the latest Growth DNA and flags fields that, if sharpened, would most improve readiness scores.

Recommendation types

  • onboarding.gap.brand_voice (low)
  • onboarding.gap.competitors (low)
  • onboarding.gap.offer_description (medium)
  • onboarding.gap.landing_page_urls (high)
  • onboarding.gap.analytics_status (high)
  • onboarding.gap.geographic_target (medium)

Complete onboarding #

Onboarding is the most important data you'll enter. Every agent reads from it. Inaccurate answers reduce signal quality across the entire system.

Fields to fill carefully

Offer description
2–3 sentences naming who it's for, what it does, and the wedge. Below 80 characters and the Onboarding-Insight agent will flag it.
Brand voice
Tone, sentence-length preference, words to avoid, whether emojis are allowed. The Creative Strategy and Content Writer agents read this verbatim.
Competitors
1–3 direct competitors with URLs. Market Intelligence builds the matrix from this list.
Analytics status
Honest answer matters. configured implies GA4 + conversion events are wired up — agents will trust your data and propose ROAS-based actions. Anything else and the Tracking agent will recommend you fix it first.

You can edit these later from workspace settings. Re-running agents picks up the new values automatically.

Connect a provider #

Connections use real OAuth flows. Tokens are encrypted at rest with Fernet; the frontend never touches an access token directly.

Steps

  1. Open Integrations in the sidebar.
  2. Find the provider card. Click Connect.
  3. You're redirected to the provider's consent screen. Approve.
  4. You land back at /integrations with the card showing status connected and the account name.

What's needed for outbound writes

Reading data only requires the standard scopes. To write back to a provider (pause campaigns, change budgets) the OAuth grant must include the provider's write scope. AdVanta refuses to dispatch writes when the recorded scopes don't carry it — you'll see a clear error and the recommendation execution row will be marked failed with reason missing write scopes.

Disconnecting

Click Disconnect. Tokens are deleted; future agent runs that would have called the provider skip cleanly. Past audit rows are preserved.

Sync logs

Each provider sync writes a row to sync_logs — visible on the integration detail page so you can see when the last successful pull happened and what the result was.

Approve a recommendation #

  1. Open Recommendations.
  2. Click into a card. The detail page shows what the agent found, why it matters, expected impact, and the suggested action.
  3. Optionally click Edit to refine the user-facing copy before approving.
  4. Click Approve.

If the recommendation carries an executable action (most paid-ads + landing-page recs do), the change is applied to the connected provider as part of the same transaction. The execution row appears at the bottom of the detail page with a snapshot diff (prior_state) and the provider's response. If the provider call fails, the execution row is marked failed — the approval still succeeds; you can retry once the underlying issue is fixed.

To revert an executed change, click Revert on the execution row. The captured prior state is replayed back to the provider; a new execution row with is_revert=true is created.

Configure Autopilot #

Autopilot is OFF by default and Owner-only. Treat it like a CI/CD release gate: never enable it without complete guardrails.

Steps

  1. Open Autopilot in the sidebar.
  2. Set every guardrail (see the table below).
  3. Switch Mode to autopilot.
  4. Click Save. Refusal with code autopilot_config_invalid means at least one guardrail is missing; the message names which.
  5. Open the Preview panel — every open recommendation shows what Autopilot would do right now, plus the rule it matched or violated.

Guardrails (every one is required for AUTOPILOT mode)

FieldWhat it does
max_daily_spend_increase_centsCaps total budget added per day across all auto-approved changes.
max_daily_spend_total_centsHard ceiling on absolute daily spend after auto-approvals.
max_pct_increase_per_changeRefuses any single change that raises a campaign budget by more than X percent.
min_conversion_thresholdA budget-up change is only allowed when the campaign's recent conversions ≥ this number.
allowed_action_typesRecommendation types eligible for auto-approval. Empty list = nothing eligible.
risk_ceilingHighest risk level eligible (low / medium / high). Recommendations above the ceiling stay manual.

Stop-loss

The Autopilot page has an emergency kill switch — when active, every auto-approval is refused with reason stop_loss_active, regardless of guardrails. Useful for incidents. Click again to clear.

Audit attribution

Auto-approvals are logged with actor_type = SYSTEM and action autopilot.approved. The matched-rules list is captured in the metadata so you can prove which guardrails passed.

Set up 2FA #

AdVanta uses TOTP-based 2FA. Use any authenticator app that supports the otpauth:// URI standard (Google Authenticator, 1Password, Authy, Bitwarden).

Enroll

  1. Open Profile in the sidebar.
  2. In the Two-factor authentication card, click Enable 2FA.
  3. The card shows a base32 secret + a provisioning URI. Add either to your authenticator app.
  4. Type the current 6-digit code into the form. Click Confirm and enable.
  5. You'll see 8 recovery codes. Save them somewhere safe — they're shown exactly once.

Sign in with 2FA on

After your password, the API returns two_factor_required. The login form reveals an OTP field; type the current code (or one of your unused recovery codes) and submit.

Recovery codes

Each recovery code works once. Used codes are removed from your account immediately. You start with 8; if you run out, disable + re-enable 2FA to get a fresh batch.

Disable

On the Profile page, type a current OTP into the disable form. The secret and recovery hashes are wiped.

Mint & use API keys #

API keys give programmatic access to a single workspace. Keys are minted by Owners only, scoped to a role floor, and the plaintext is shown exactly once.

Mint

  1. Open API keys in the sidebar.
  2. Click Mint key. Provide a label (e.g. "ci-ingest", "zapier"). Pick the role.
  3. Copy the plaintext value. It looks like ak_xxxxxxxx.yyyyyyyy…. Refreshing the page makes it unrecoverable.

Use

curl https://<your-deploy>/api/v1/workspaces/<ws_id>/agents \
  -H "Authorization: ApiKey ak_xxxxxxxx.yyyyyyyy"

Any endpoint a normal session can hit, the API key can hit too — subject to the role floor you set when minting. The key's last_used_at timestamp updates within seconds of use.

Revoke

On the API keys page, click Revoke. Subsequent calls return 401 within a few seconds.

Cross-workspace requests are refused

A key minted in workspace A used against workspace B's URL returns 404 workspace_not_found. The same applies even if the user who minted the key is a member of both — the key is scoped to one workspace.

Write & publish a blog post #

AdVanta ships a full blog platform inside the dashboard. The same surface customers use to draft + publish becomes the public archive at /blog.

Create a post

  1. Open Blog in the sidebar (route: /blog/posts).
  2. Click New post. Enter a working title. Click Create + edit.
  3. You land on the editor at /blog/posts/{id}.

Editor tab

Title field on top, body (markdown) below, live preview on the right. Two ways to add images:

  • Drag-drop any image file (JPEG, PNG, WebP, GIF — under 5 MB) into the body textarea. It uploads + an image markdown line is inserted at the cursor.
  • Click Insert image in the Compose card header to use a file picker.

The preview supports headings, ordered + unordered lists, bold, italic, inline code, code blocks, links, and images.

AI Assistant tab

Five action buttons, each backed by the workspace's LLM budget. With no OPENAI_API_KEY set, you get a clearly-tagged deterministic stub instead of a hallucination.

ActionUse when
OutlineYou need a section structure for the post. Returns 5–8 headings + 1-line summaries.
Expand selectionPaste a heading or bullet into Selection. Returns a paragraph (≤180 words).
Refine selectionRewrite the selected passage for clarity / tone / length.
Suggest titleReturns 5 candidate titles ≤ 70 chars.
Suggest metaReturns meta_title (≤60) + meta_description (≤155).

Each suggestion renders as a card. Use Append to body, Use first as title, Apply to settings, or Discard. Nothing is auto-applied — you stay in control.

Settings tab

Slug, excerpt, target URL, comma-separated keywords, cover image, plus full SEO metadata (meta_title, meta_description, og_image, canonical_url). Slug is normalized (lowercase, hyphenated) and unique within your workspace.

Approve → Publish

From the editor header: Approve moves the draft to approved; Publish moves it to published and sets published_at. If the slug or excerpt was blank, publish auto-fills both — slug from the title, excerpt from the body's first ~280 characters.

The public archive

To make published posts visible at the public marketing URL /blog, the platform admin needs to set MARKETING_WORKSPACE_SLUG to your workspace slug. Until then, /blog shows a coming-soon placeholder (no fabricated posts).

Run an A/B test #

Create

  1. Open A/B tests in the sidebar. Click New test.
  2. Pick the target type:
    • Landing page — variants are URLs. Visitors get assigned via the public traffic-split snippet (/static/advanta-ab.js).
    • Ad — variants become campaigns on a connected provider when you launch.
  3. Add at least 2 variants with traffic shares that sum to 1.0. (Default is even split.)

Generate variants with AI

On the test detail page, click Generate variants. The generator creates 1–4 new variants from your control payload and rebalances traffic shares so the total still sums to 1.0.

Launch

Landing-page tests launch instantly (no provider write — just timestamp variants). Ad tests call the provider once per variant; partial failures keep the test in READY so you can retry.

Record metrics + declare a winner

  1. For landing-page tests, conversions stream in via the public snippet automatically.
  2. For ad tests, sync metrics from GA4 (Sync from GA4 on the detail page) or paste manual numbers.
  3. Click Declare winner on the leading variant. The two-proportion z-test must pass (p < 0.05) and each variant must meet the minimum sample size — the API refuses with ab_test_underpowered or ab_test_inconclusive otherwise. Pass force=true only when you have off-platform context that justifies it.

Bandit strategies

static: each variant gets its declared traffic share for the test's lifetime.
thompson_sampling: traffic dynamically funnels toward the best-performing variant (Beta posterior, sampled per assignment) while still exploring others.

Billing #

The Billing page (/billing) shows your active plan, the live subscription status, and a usage strip with rolling 30-day counters:

  • Agent runs vs. plan cap.
  • Landing pages on file.
  • Team members.
  • LLM dollar spend (see below).

LLM dollar cost #

Every LLM call records both the token total (for plan caps) and an estimated USD cost (for visibility). The cost is computed from a per-model rate table at call time — no recompute, no nightly rollup.

The Billing page shows the rolling-30-day total in dollars + tokens. If you want a finer-grained view, query the usage_events table with event_type=LLM_CALL; the per-call estimate lives in metadata.estimated_cost_usd_micros.

Admin console #

The admin console at /admin is gated to users with is_superuser=true on the User row — set manually via SQL on the server. It's not exposed via the UI.

It surfaces system-wide:

  • 18 counter tiles: users, superusers, workspaces, paid workspaces, agent runs (total + 7d), open recs, integrations connected, landing pages, reports, executions, content drafts, drafts published, outreach sent, prospects, A/B tests active + done, etc.
  • Workspaces table — newest first, with member count + plan.
  • Users table — every active account with its workspace count.

Security model #

Authentication

  • JWT access tokens (30-minute default lifetime, configurable via JWT_ACCESS_TOKEN_EXPIRE_MINUTES).
  • HTTP-only refresh-token cookie scoped to /api/v1/auth, signed, 30-day default.
  • Passwords hashed with bcrypt.
  • Optional TOTP 2FA per user.
  • Optional Google OAuth login (no extra scopes — login only).

Authorization

  • Workspace isolation enforced at every endpoint via get_current_member — a request whose path workspace doesn't match a membership returns 404, never a 403 (which would confirm existence).
  • RBAC: 5 roles, ranked. require_role_at_least guards each mutation.
  • API keys: workspace-scoped, role-floored, never returned in plaintext after creation.

Token storage

OAuth tokens for connected providers (Google Ads, Meta, LinkedIn, GA4, Search Console, Stripe) are encrypted at rest with Fernet using ENCRYPTION_KEY. The frontend never sees a provider access token.

Audit log

Append-only. See Audit log for fields. Includes IP and User-Agent on every row.

Rate limiting

Per-workspace + per-route limits enforced via Redis. When Redis isn't reachable the middleware fails open with a warning log so the app stays usable in dev.

Webhook signatures

Inbound provider webhooks are verified with HMAC-SHA256 against per-provider shared secrets. Stripe webhooks use Stripe's standard signature scheme. See Provider webhooks.

API reference #

The full machine-readable spec lives at /api/v1/docs (Swagger UI) and /api/v1/openapi.json. This section captures the conventions.

Authentication

Every authenticated endpoint accepts one of:

  • Authorization: Bearer <jwt_access_token> — for human session calls.
  • Authorization: ApiKey ak_<prefix>.<secret> — for programmatic calls.

Common request headers

  • Content-Type: application/json — for JSON bodies.
  • X-Provider-Signature: <hex> — for provider webhooks.
  • X-Inbound-Secret: <shared> — for inbound email webhooks.
  • Idempotency-Key: <your-key> — supported on recommendation execution endpoints to prevent duplicate provider writes.

Error envelope

Every non-2xx response carries:

{
  "error": {
    "code": "machine_readable_code",
    "message": "Human-readable explanation",
    "request_id": "uuid-v4"
  }
}

The request_id is also returned in the X-Request-Id response header — quote it when reporting bugs.

Verification

Provider webhooks (non-Stripe) are signed with HMAC-SHA256 over the raw body:

signature = hmac_sha256(SECRET, raw_body).hex()
# Header may be either:
#   X-Provider-Signature: <hex>
#   X-Provider-Signature: sha256=<hex>

Stripe uses the standard Stripe-Signature header (timestamp + v1 signature) — verified by Stripe's own SDK in our handler.

Behavior on unrecognized accounts

When a webhook authenticates correctly but the account_id doesn't match any connected account in any workspace, the request is 200-acked with {"matched": false, "reason": "unrecognized_account"}. This prevents providers from retrying forever for accounts that have been disconnected.

FAQ #

Does AdVanta actually change my campaigns, or just suggest things?

Both, on your terms. By default every recommendation is suggestion-only until you approve it. Once approved, the change is applied to the connected provider — with prior state captured so any single write is reversible. Autopilot Mode auto-approves recs that pass every guardrail you set; until you turn it on, nothing ships without a human.

Where do my OAuth tokens live?

Encrypted at rest with Fernet, never exposed to the frontend, refreshed automatically when the provider supports it. Provider calls happen server-side; your browser never touches a provider access token.

Is the AI looking at my private data?

Only the workspace data you connect, and only when an agent is running for that workspace. LLM calls are scoped per-prompt — no cross-customer training, no embedding storage. We don't sell data.

What happens if I disconnect a provider?

Sync stops, recommendations for that platform freeze, and dashboards fall back to honest empty states with a connect button. Tokens are deleted. Audit history is preserved.

What's the difference between Approval and Autopilot?

Approval requires a human click on every change. Autopilot auto-approves changes that pass every guardrail you've set (spend cap, % cap, conversion threshold, allowed action types, risk ceiling). Autopilot is OFF by default and Owner-only to enable.

Why is the AI Assistant showing "stub (no LLM key)"?

Your deployment doesn't have OPENAI_API_KEY set. The flow still works for testing — you'll get a clearly-tagged deterministic placeholder instead of an LLM-generated suggestion. Ask your admin to set the key.

Why does the public blog at /blog say "Coming soon"?

The deployment hasn't set MARKETING_WORKSPACE_SLUG to point at one of your workspaces. Until it does, /blog shows the placeholder — never fabricated posts. Set the env var to your workspace slug and restart the API.

Can I export my data?

Yes. CSV exports for prospects, content drafts, and recommendation executions live under /workspaces/{ws}/<resource>.csv. Reports download as multi-section CSV or branded PDF.

How do I delete my workspace?

Contact privacy@advantaai.com. Workspace deletes are processed within 30 days; encrypted backups age out within 90 days.

Glossary #

Agent
A specialist piece of growth expertise that runs against your workspace data and produces tasks, skill outputs, and recommendations.
AgentRun
One invocation of one agent. Has status (running / succeeded / failed), input + output payloads, model + token counts, and a triggering user.
AGE / GEO

Append-only record of every meaningful action in a workspace — approvals, executions, autopilot decisions, webhook events, key mints, 2FA changes. See Audit log.
Autopilot
Mode in which AdVanta auto-approves recommendations that pass every guardrail. Off by default; Owner-only to enable.
ContentDraft
The shared record type behind every piece of generated content — blog posts, ad copy, landing-page copy, emails, social posts. Has a status lifecycle (draft → approved → published / rejected / archived).
Creative
An ad-copy variant. Produced by the Creative Strategy agent (source = ai_generated) or pulled from a connected platform (source = platform_synced).
Execution
The act of applying an approved recommendation to a connected provider. Captures prior state so it can be reverted.
Growth DNA Profile
A scored snapshot of a workspace's growth readiness — funnel, paid ads — plus a 30-day plan. Generated from the onboarding profile.
Recommendation
A user-facing suggestion produced by an agent. Carries a risk level and an approval flow.
Skill output
Structured JSON an agent persisted as part of a run — used by reports, the dashboard, and downstream agents.
Workspace
The top-level tenant. Every record belongs to one. Members get role-gated access.
